F5: Accessibility
Accessibility ensures every customer can shop independently by implementing WCAG 2.1 AA standards, 
keyboard navigation, and screen reader support. This unlocks an underserved market of 1.3 billion
people with disabilities while improving the experience for all users through universal design
principles. Features built for accessibility benefit everyone, from mobile users to those seeking
clearer navigation, positioning us as an ethical industry leader while making our platform fundamentally
better for all customers.


F5.1: As a user with motor impairments, I want full keyboard navigation so I can shop without a mouse.
• All interactive elements accessible via Tab key
• Logical tab order follows visual layout
• Skip navigation link at page top
• Focus indicators clearly visible (2px solid outline)
• No keyboard traps (user can always navigate away)
• Dropdown menus accessible via Arrow keys
• Esc key closes modals and overlays

F5.2: As a blind user, I want screen reader compatibility so I can shop independently.
• All images have descriptive alt text
• Form fields have associated labels (not just placeholders)
• ARIA landmarks used (nav, main, aside, footer)
• ARIA live regions for dynamic content updates
• Proper heading hierarchy (h1, h2, h3)
• Link text is descriptive (not just "click here")
• Product prices and ratings announced correctly

F5.3: As a user with visual impairment, I want high color contrast so I can read all text clearly.
• Text contrast ratio minimum 4.5:1 (normal text) / 3:1 (large text)
• Color not used as sole indicator (use icons + color)
• Interactive elements have minimum 44x44px touch target size
• Text resizable up to 200% without breaking layout
• No information loss when zooming to 400%
• Focus indicators meet 3:1 contrast ratio

F5.4: As a user with cognitive impairments, I want accessible forms so I can complete orders successfully.
• Form fields have clear, descriptive labels
• Required fields marked with asterisk and "required" text
• Input format examples shown (e.g., "MM/DD/YYYY")
• Error messages specific and actionable
• Errors announced to screen readers immediately
• Error summary at top of form with links to fields
• Timeout warnings with option to extend session

F5.5: As an accessibility specialist, I want regular a11y audits so we ensure WCAG compliance.
• Automated testing with aXe, Lighthouse, WAVE
• Manual testing with NVDA, JAWS, VoiceOver screen readers
• Keyboard-only navigation testing
• Color contrast analysis tools
• User testing with people with disabilities
• Accessibility statement page published
• WCAG 2.1 AA compliance badge displayed
